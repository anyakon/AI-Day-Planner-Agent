# Product Proposal
AI Day Planner Agent

## 1. Обоснование

Планирование дня — рутинная задача. Люди тратят время на расстановку задач, часто создают конфликты в расписании и забывают о дедлайнах.

Агент автоматизирует этот процесс: принимает текстовый запрос на естественном языке ("Завтра утром теннис, после обеда код-ревью"), распознаёт задачи, проверяет календарь, размещает задачи без пересечений и создаёт события.

Архитектура: ReAct (Reason + Act) через LangGraph State Graph. LLM сама решает какие инструменты вызвать, анализирует результаты и повторяет цикл.

## 2. Цель и метрики

### Цель

Принимать текстовые запросы пользователя, автоматически распознавать задачи и размещать их в календаре без конфликтов.

### Продуктовые метрики

- task_placement_rate >= 90% (доля задач которые удалось разместить)
- success_rate >= 90% (доля успешных запусков)
- avg_plan_quality >= 0.7 (общий скор evals)
- response_time < 60 секунд для текстового запроса (NLP + агент)

### Агентные метрики

- корректность распознавания NLP (все задачи извлечены)
- соблюдение suggested_start (утро=09:00, день=13:00, вечер=18:00)
- соблюдение дедлайнов
- отсутствие пересечений с существующими событиями
- time_to_first_token (p50/p95)
- time_per_output_token (p50/p95)
- guardrail_rejections

### Технические метрики

- p95 e2e latency < 60 секунд
- успешность вызова инструментов > 95%
- rate limit не превышается
- cost per run < $0.02

## 3. Сценарии

### Сценарий 1: Текстовый запрос

Пользователь пишет: "Завтра утром теннис, после обеда код-ревью". Агент:
1. Проверяет guardrails (PII, injection)
2. NLP парсер извлекает задачи с suggested_start
3. Анонимизирует задачи
4. Запускает ReAct цикл
5. Создаёт события

### Сценарий 2: Планирование на несколько дней

Если задачи не помещаются сегодня, агент автоматически планирует их на завтра или другие дни через get_date(offset_days).

### Сценарий 3: Конфликты в календаре

Агент проверяет существующие события и не создаёт пересечений. Если время занято — выбирает ближайшее свободное после suggested_start.

## Edge cases

- Все задачи в прошлом — агент предлагает перенести
- Дедлайн раньше текущего времени — задача откладывается
- Календарь полностью занят — задачи размещаются в ближайшее свободное время
- Нет доступа к календарю — агент работает без проверки событий
- LLM недоступна — агент завершает работу с ошибкой
- Prompt injection detected — guardrail rejection, запрос отклонён
- Ночное время — запрет на 00:00-07:00

## 4. Ограничения

### Технические

- Требуется LLM (нет алгоритмического fallback)
- Лимиты LLM API по токенам и времени
- Google Calendar OAuth требует настройки для каждого пользователя
- Rate limit: 10 вызовов инструментов в минуту
- Двойной вызов LLM (NLP парсинг + агент)

### Операционные

- Максимум 25 итераций ReAct-цикла
- События нельзя создавать в прошлом
- События нельзя создавать без валидации
- Max 2 попытки валидации перед созданием
- Запрет на ночное время (00:00-07:00)

## 5. Архитектура

```
Текст пользователя
       │
       ▼
┌──────────────┐
│ NLP Parser   │  LLM: извлекает задачи с suggested_start
└──────┬───────┘
       ▼
┌──────────────┐
│ Guardrails   │  Pydantic + injection check + PII
└──────┬───────┘
       ▼
┌──────────────┐
│ Agent (ReAct)│  LangGraph: think → execute
└──────┬───────┘
       ▼
┌──────────────┐
│ Calendar     │  Google Calendar API
│ + Monitoring │  Prometheus, Grafana, Langfuse
└──────────────┘
```

### Модули

- `planner.py` — агент, LangGraph, 5 инструментов
- `nlp_parser.py` — распознавание задач из текста
- `config.py` — конфигурация
- `security.py` — анонимизация PII, валидация, RateLimiter
- `guardrails.py` — injection detection, Pydantic, output safety
- `evals.py` — оценка качества (4 метрики + overall)
- `monitoring.py` — MetricsCollector, LangfuseTracker, Prometheus format
- `server.py` — FastAPI API, Prometheus endpoint

## 6. Data Flow

1. Пользователь отправляет текст через POST /api/chat
2. Guardrail check: PII, injection, валидация
3. NLP парсер извлекает задачи (LLM + suggested_start)
4. Анонимизация задач (security.py)
5. ReAct цикл:
   - LLM решает вызвать инструмент
   - Выполнение инструмента с rate limiting
   - LLM анализирует результат
6. После завершения — оценка качества (evals.py)
7. Сохранение метрик: latency, tokens, cost (monitoring.py)
8. Langfuse trace
9. Возврат плана пользователю
10. Prometheus scrape каждые 10 секунд
11. Grafana визуализация

### Что делает LLM

- NLP: распознаёт задачи из текста
- Решает какие инструменты вызвать
- Анализирует результаты
- Составляет план задач
- Объясняет решения

### Что НЕ делает LLM

- Не проверяет пересечения напрямую (validate_schedule)
- Не работает с API напрямую (create_event, get_events)
- Не хранит данные
- Не обрабатывает PII (анонимизируется до отправки)

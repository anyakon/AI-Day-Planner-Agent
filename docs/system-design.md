# Архитектура AI-Day-Planner-Agent (PoC)

## Ключевые архитектурные решения
- **Паттерн агента:** State Graph (на базе LangGraph) для реализации циклов рефлексии (проверка расписания на накладки перед финальным ответом)
- **LLM Gateway:** Единая точка доступа к LLM с паттерном Circuit Breaker и Fallback (переключение с OpenAI GPT-4o на Claude 3.5 Haiku при недоступности)
- **State & Memory:** PostgreSQL (или SQLite для PoC) для хранения долгосрочных предпочтений пользователя и Redis для кратковременной памяти сессии и кэширования ответов API.
- **Интеграция с API (Tools):** Выполнение тулов (Google Calendar API, Weather) в изолированном слое с таймаутами и проверкой на идемпотентность (защита от создания дублей событий).

## Список модулей и их роли
1. **API / Webhook Gateway:** Прием запросов (Telegram / REST API), аутентификация, валидация лимитов (Rate Limiting).
2. **Orchestrator (Agent Core):** Управление графом состояний (LangGraph), принятие решений о вызове тулов, генерация плана.
3. **Memory & Context Manager:** Управление контекстным окном, инъекция текущего времени/даты, загрузка правил планирования (preferences).
4. **Tool Execution Engine:** Безопасный вызов внешних API с обработкой ошибок и ретраями.
5. **Observability Logger:** Асинхронная отправка трейсов, метрик токенов и логов ошибок.

## Основной Workflow выполнения задачи
1. **User Input:** Пользователь пишет: *"Спланируй завтра. Утром тренировка, днем 2 созвона, вечером в кино".*
2. **Context Enrichment:** Подтягивается текущая дата, часовой пояс пользователя, его глобальные предпочтения ("не ставить встречи до 10:00").
3. **Calendar Check:** Агент вызывает тул `check_calendar` для получения уже существующих событий на завтра.
4. **Draft Planning (LLM):** Агент составляет черновик расписания.
5. **Validation (Guardrails):** Внутренний шаг валидации. Проверка: нет ли накладок? Учтено ли время на дорогу? 
6. **Execution:** Если расписание валидно, агент вызывает `create_calendar_events`.
7. **Response:** Формирование красивого ответа пользователю со ссылками на события.

## Описание State / Memory / Context Handling
- **Session State (Краткосрочная):** Хранится в рамках текущего графа выполнения (LangGraph thread_id). Очищается при завершении планирования дня.
- **Long-term Memory (Долгосрочная):** Векторная БД (Chroma/FAISS для PoC) или JSON-B в Postgres. Хранит факты вида: *"Пользователь бегает по 40 минут", "Обед обычно в 14:00"*.
- **Context Budgeting:** Строгое ограничение: не более N токенов на историю чата, не более M токенов на системный промпт и правила. Используется алгоритм вытеснения (Rolling Window). 

## Retrieval-контур
- **Источники:** Явные настройки профиля + извлеченные агентом предпочтения из прошлых бесед.
- **Процесс:** При каждом запросе на планирование делается similarity search по эмбеддингам запроса, чтобы достать релевантные привычки.

## Tool / API интеграции
- **Google Calendar API:** Контракты на чтение и запись.
- **Weather API:** Для проверки погоды, если запланирована активность на улице.
- **Routing API (Google Maps):** Оценка времени на дорогу между оффлайн-событиями.

## Failure Modes, Fallbacks и Guardrails
| Угроза / Сбой | Guardrail / Fallback механизм |
| :--- | :--- |
| **LLM API Timeout/502** | Повтор (Retry) с экспоненциальной задержкой. При 3 неудачах — фоллбэк на запасную модель (например, OpenAI -> Anthropic). |
| **Галлюцинация времени** | Валидатор проверяет, что `start_time < end_time` и события не пересекаются (Self-Correction loop). |
| **Превышение Context Window** | Автоматическая суммаризация старых сообщений сессии. |
| **Calendar API Rate Limit** | Кэширование запросов на чтение (Redis). При записи — откладывание задачи в очередь. |
| **Опасные действия LLM** | Тул `delete_event` отключен на уровне PoC (Hard constraint). Разрешено только создание. |

## Ограничения (Constraints)
- **Latency:** Ответ пользователю с планом < 15 секунд (с учетом вызова API календаря).
- **Cost:** < $0.02 на одну сессию планирования (контроль через мониторинг токенов).
- **Reliability:** Uptime сервиса 99%, доступность LLM-контура маскируется ретраями.
